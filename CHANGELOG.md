## Pipeline fixes — 2026-05-28

- **PR #147 — Fix observation-time bucketing in ownership_completions (Cause C)** (GitHub #147, merge `9cf4f8a`): pipeline fix for missing-hour gaps on the Hourly Output heatmap. ROOT CAUSE: `completion_detector.py` stamped `completed_at` with the detector's clock at first observation, so when polls failed and a later run recovered, large batches were bulk-stamped into a single microsecond, emptying the true hour and dumping the work into the next one (e.g. 493 records sharing `2026-05-27T15:24:05.888033Z` blanked EAT17 yesterday; 955 records sharing `2026-05-28T12:55:17.148266Z` blanked EAT14 today). FIX: new `resolve_completion_ts(state, raw_payload, fallback)` derives the real completion time from the Airtable date field matching each record's state — `valid_selected_time` for Valid, `done_selected_time` for Done, `last_modified` for tagged/need-to-be-update/Selected-for-BO-QA (start_tagging_date proven too coarse — it's a batch-assignment field that clusters whole tasks into single minutes). `detected_at` retains observation time for ordering/idempotency. Handles both legacy + REST Airtable field-ID schemes (raw_payload uses either depending on source_table — the IO→support remap deliberately doesn't cover date fields). Plus a stand-alone `.poll_work/backfill_completed_at.py` for one-off correction of historical rows, with `--apply` flag (idempotent, batched, dry-run by default). 13 new unit tests cover legacy/REST/missing-field/state-specific resolution. Live verification: 2026-05-27 backfill applied (5,628 rows patched, EAT17 recovered 0 → 308 matching dry-run prediction exactly); 2026-05-28 backfill applied (6,732 rows patched, EAT14 recovered 0 → 483); dashboards verified via real-boot CDP harness on both dates; post-merge cron runs on the new SHA writing correct per-record times.

- **PR #148 — Backfill: compare timestamp instants, not strings (fixes idempotency churn)** (GitHub #148, merge `0017d89`): follow-up to PR #147. The backfill compared raw timestamp strings to decide if a row needs an update — but Postgres normalizes `2026-05-28T17:52:55.000Z` to `2026-05-28T17:52:55+00:00`, so 228 rows showed up flagged forever despite being at the correct instant (same instant, different text). FIX: new `_same_instant` / `_needs_update` helpers parse both timestamps and compare with a 1-second tolerance, replacing string equality at the patch gate. `min_cluster` / `drift_minutes` retained for CLI compatibility + dry-run diagnostics but no longer gate. 12 new unit tests cover format-only equality, sub-second / minute / hour drift, and missing-stored-value cases. Final cleanup sweep over 2026-05-27..29 (post-merge): 91 genuine sub-30min stragglers found (which the OLD cluster-OR-drift heuristic had silently skipped) → applied → final idempotency converges to **0 flagged** for the first time. Target hours stable: EAT17 = 308, EAT14 = 483.
